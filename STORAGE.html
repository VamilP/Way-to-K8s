ğŸš¨ Why Storage is Hard in Kubernetes

The Challenge

Pods are ephemeral:

  • Pods can be deleted and recreated
  • Pod IPs change
  • Local storage is lost when Pod is deleted

Problem: How to persist data across Pod restarts?

The Solution: Persistent Storage

Kubernetes provides persistent storage that:

  • âœ… Survives Pod restarts
  • âœ… Can be shared between Pods
  • âœ… Managed independently of Pod lifecycle

ğŸ“¦ Volume Types Overview

1. Ephemeral Volumes (Pod Lifecycle)

Lifecycle: Tied to Pod. Deleted when Pod is deleted.

Types:

  • emptyDir - Empty directory
  • configMap - Mount ConfigMap as volume
  • secret - Mount Secret as volume

2. Persistent Volumes (Cluster Lifecycle)

Lifecycle: Independent of Pod. Survives Pod deletion.

Types:

  • PersistentVolume (PV) - Cluster resource
  • PersistentVolumeClaim (PVC) - User request for storage
  • StorageClass - Dynamic provisioning

ğŸ’¿ Persistent Volumes (PV)

What is a PersistentVolume?

PV = Cluster-wide storage resource provisioned by administrator.

Characteristics:

  • Cluster resource (like Node)
  • Has lifecycle independent of Pods
  • Can be statically or dynamically provisioned

PV Example (Static Provisioning) 

apiVersion: v1
kind: PersistentVolume
metadata:
  name: pv-1
spec:
  capacity:
    storage: 10Gi
  accessModes:
    - ReadWriteOnce
  persistentVolumeReclaimPolicy: Retain
  storageClassName: manual
  hostPath:
    path: /data/pv-1

Access Modes:

  • ReadWriteOnce (RWO): Single node read-write
  • ReadOnlyMany (ROX): Multiple nodes read-only
  • ReadWriteMany (RWX): Multiple nodes read-write

Reclaim Policies:

  • Retain: Manual cleanup (data preserved)
  • Recycle: Basic cleanup (deprecated)
  • Delete: Volume deleted automatically

ğŸ“ PersistentVolumeClaims (PVC)

What is a PersistentVolumeClaim?

PVC = User's request for storage (like a "ticket" for storage).

Purpose:

  • Request storage without knowing details
  • Kubernetes finds/binds matching PV
  • Pods use PVCs, not PVs directly

PVC Example 

apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-pvc
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 5Gi
  storageClassName: fast-ssd

ğŸ­ StorageClasses: Dynamic Provisioning

The Problem with Static Provisioning

Static: Admin creates PVs manually â†’ Time-consuming, inflexible

The Solution: StorageClasses

StorageClass = Template for dynamic PV creation.

How It Works:

PVC Created â†’ StorageClass â†’ Provisioner â†’ PV Created â†’ Bound to PVC

StorageClass Example 

apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast-ssd
provisioner: kubernetes.io/gce-pd
parameters:
  type: pd-ssd
  replication-type: regional-pd
volumeBindingMode: WaitForFirstConsumer
allowVolumeExpansion: true

Common Provisioners:

  • kubernetes.io/gce-pd - GCP Persistent Disk
  • kubernetes.io/aws-ebs - AWS EBS
  • kubernetes.io/azure-disk - Azure Disk
  • kubernetes.io/vsphere-volume - vSphere
  • kubernetes.io/portworx-volume - Portworx
  • kubernetes.io/cephfs - CephFS

Volume Binding Modes:

  • Immediate: PV created immediately
  • WaitForFirstConsumer: PV created when Pod is scheduled

  name: mysql

Lab 1: Create PVC and Use in Pod

Create PVC:

apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-pvc
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 1Gi

Create Pod:

apiVersion: v1
kind: Pod
metadata:
  name: test-pod
spec:
  containers:
  - name: app
    image: nginx
    volumeMounts:
    - name: data
      mountPath: /data
  volumes:
  - name: data
    persistentVolumeClaim:
      claimName: my-pvc

Apply:

kubectl apply -f pvc.yaml
kubectl apply -f pod.yaml

# Check PVC status
kubectl get pvc

# Check PV (if dynamically created)
kubectl get pv

# Write data
kubectl exec test-pod -- sh -c "echo 'Hello' > /data/test.txt"

# Delete Pod
kubectl delete pod test-pod

# Recreate Pod (data persists)
kubectl apply -f pod.yaml
kubectl exec test-pod -- cat /data/test.txt

Lab 2: StatefulSet with PVC Template

Create StatefulSet:

apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: web
spec:
  serviceName: web
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.21
        volumeMounts:
        - name: www
          mountPath: /usr/share/nginx/html
  volumeClaimTemplates:
  - metadata:
      name: www
    spec:
      accessModes: ["ReadWriteOnce"]
      resources:
        requests:
          storage: 1Gi

Apply:

kubectl apply -f statefulset.yaml

# Check PVCs (one per Pod)
kubectl get pvc
# Output: www-web-0, www-web-1, www-web-2

# Check Pods
kubectl get pods -l app=nginx

ğŸ› Troubleshooting Storage

PVC Stuck in Pending 

# Check PVC
kubectl describe pvc <pvc-name>

# Common issues:
# - No StorageClass specified and no default
# - No available PVs (static provisioning)
# - Provisioner not working (dynamic provisioning)

PV Not Bound 

# Check PV
kubectl describe pv <pv-name>

# Common issues:
# - Access modes don't match
# - Storage capacity insufficient
# - StorageClass mismatch

Pod Can't Mount Volume 

# Check Pod events
kubectl describe pod <pod-name>

# Common issues:
# - PVC doesn't exist
# - PVC not bound
# - Node doesn't support volume type

ğŸ“ Key Takeaways

  1. Ephemeral volumes = Pod lifecycle (emptyDir, configMap, secret)
  2. PersistentVolumes (PV) = Cluster storage resource
  3. PersistentVolumeClaims (PVC) = User request for storage
  4. StorageClasses = Template for dynamic PV provisioning
  5. PV-PVC binding = Automatic matching based on requirements
  6. StatefulSets = Use volumeClaimTemplates for per-Pod storage
  7. Access modes = RWO, ROX, RWX
  8. Reclaim policies = Retain, Delete
  9. Dynamic provisioning = PVs created automatically from StorageClass
  10. Storage survives Pod deletion = Key benefit of persistent storage
