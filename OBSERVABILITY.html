ğŸ” The Three Pillars of Observability

  • Debugging errors

ğŸ“ Logs

Viewing Pod Logs

Basic:

# View logs
kubectl logs <pod-name>

# Follow logs (like tail -f)
kubectl logs -f <pod-name>

# Previous container instance (if crashed)
kubectl logs <pod-name> --previous

# Specific container (multi-container Pod)
kubectl logs <pod-name> -c <container-name>

# Last N lines
kubectl logs <pod-name> --tail=100

# Since specific time
kubectl logs <pod-name> --since=1h

Logs from Multiple Pods 

# All Pods with label
kubectl logs -l app=nginx

# All Pods in Deployment
kubectl logs -f deployment/nginx-deployment

Log Locations

Container Logs:

  • Stored on node: /var/log/pods/
  • Managed by kubelet
  • Rotated automatically

System Logs:

  • kubelet: journalctl -u kubelet
  • kube-proxy: Pod logs in kube-system
  • API server: Pod logs in kube-system

ğŸ“Š Events

What are Events?

Events = Records of what happened in the cluster.

Types:

  • Normal (informational)
  • Warning (issues)

Viewing Events 

# All events in namespace
kubectl get events

# Watch events
kubectl get events --watch

# Events for specific resource
kubectl describe pod <pod-name> | grep -A 10 Events

# Events sorted by time
kubectl get events --sort-by='.lastTimestamp'

Event Fields

  • Type: Normal, Warning
  • Reason: Why event occurred
  • Message: Human-readable description
  • Source: Component that created event
  • First/Last Seen: Timestamps

Common Event Reasons

  • Failed - Pod failed to start
  • BackOff - Container restarting (backoff)
  • Unhealthy - Health check failed
  • Scheduled - Pod scheduled to node
  • Pulled - Image pulled
  • Created - Container created
  • Started - Container started

ğŸ”§ kubectl Debugging Commands

describe

Purpose: Detailed information about resource.

# Describe Pod
kubectl describe pod <pod-name>

# Describe Node
kubectl describe node <node-name>

# Describe Service
kubectl describe svc <service-name>

What it shows:

  • Resource spec
  • Current status
  • Events
  • Conditions
  • Resource usage

get with -o wide/yaml/json 

# Wide output (more columns)
kubectl get pods -o wide

# YAML output
kubectl get pod <pod-name> -o yaml

# JSON output
kubectl get pod <pod-name> -o json

# JSONPath
kubectl get pods -o jsonpath='{.items[*].metadata.name}'

exec

Purpose: Execute command in running container.

# Execute command
kubectl exec <pod-name> -- ls /app

# Interactive shell
kubectl exec -it <pod-name> -- /bin/bash

# Specific container
kubectl exec -it <pod-name> -c <container-name> -- /bin/bash

port-forward

Purpose: Forward local port to Pod port.

# Forward port
kubectl port-forward <pod-name> 8080:80

# Forward to Service
kubectl port-forward svc/<service-name> 8080:80

# Specific namespace
kubectl port-forward -n <namespace> <pod-name> 8080:80

top

Purpose: Resource usage (CPU, memory).

# Pod resource usage
kubectl top pods

# Node resource usage
kubectl top nodes

# Specific namespace
kubectl top pods -n <namespace>

# Requires metrics-server installed

ğŸ› Common Debugging Scenarios

1. Pod Crash Loop

Symptoms:

  • Pod status: CrashLoopBackOff
  • Pod keeps restarting

Debug Steps:

# Check Pod status
kubectl get pods

# Describe Pod
kubectl describe pod <pod-name>

# Check logs
kubectl logs <pod-name>

# Previous instance logs
kubectl logs <pod-name> --previous

# Check events
kubectl get events --field-selector involvedObject.name=<pod-name>

Common Causes:

  • Application error
  • Wrong image
  • Missing environment variables
  • Resource limits exceeded
  • Health check failing

2. Pod Pending

Symptoms:

  • Pod status: Pending
  • Not scheduled to node

Debug Steps:

# Describe Pod
kubectl describe pod <pod-name>

# Check events
kubectl describe pod <pod-name> | grep -A 10 Events

# Check node resources
kubectl describe nodes

# Check node selectors/affinity
kubectl get pod <pod-name> -o yaml | grep -A 10 nodeSelector

Common Causes:

  • Insufficient resources
  • Node selector mismatch
  • Taints not tolerated
  • Affinity rules not satisfied

3. Pod Not Ready

Symptoms:

  • Pod status: Running but not Ready
  • Readiness probe failing

Debug Steps:

# Check Pod status
kubectl get pods

# Describe Pod
kubectl describe pod <pod-name>

# Check readiness probe
kubectl get pod <pod-name> -o yaml | grep -A 10 readinessProbe

# Test endpoint manually
kubectl exec <pod-name> -- curl http://localhost:8080/health

Common Causes:

  • Readiness probe path wrong
  • Application not ready
  • Port mismatch

4. Service Not Working

Symptoms:

  • Can't reach Service
  • Connection refused

Debug Steps:

# Check Service
kubectl get svc <service-name>
kubectl describe svc <service-name>

# Check Endpoints
kubectl get endpoints <service-name>

# Check Pod labels match selector
kubectl get pods --show-labels
kubectl get svc <service-name> -o yaml | grep selector

# Test from Pod
kubectl run test --image=curlimages/curl -it --rm -- sh
# Inside: curl http://<service-name>

Common Causes:

  • No Pods match selector
  • Port mismatch
  • Pods not ready

5. Networking Issues

Symptoms:

  • Pods can't communicate
  • DNS not resolving

Debug Steps:

# Check Pod IPs
kubectl get pods -o wide

# Test connectivity
kubectl exec <pod-1> -- ping <pod-2-ip>

# Check DNS
kubectl run dns-test --image=busybox -it --rm -- nslookup <service-name>

# Check CoreDNS
kubectl get pods -n kube-system | grep coredns
kubectl logs -n kube-system <coredns-pod>

6. Image Pull Errors

Symptoms:

  • Pod status: ImagePullBackOff or ErrImagePull

Debug Steps:

# Describe Pod
kubectl describe pod <pod-name>

# Check image
kubectl get pod <pod-name> -o jsonpath='{.spec.containers[*].image}'

# Test image pull manually
docker pull <image-name>

Common Causes:

  • Image doesn't exist
  • Wrong image name/tag
  • Private registry authentication missing
  • Network issues

ğŸ” Debugging Workflow

Step-by-Step Process 

1. Check resource status
   kubectl get <resource>

2. Get detailed information
   kubectl describe <resource>

3. Check logs
   kubectl logs <pod-name>

4. Check events
   kubectl get events

5. Verify configuration
   kubectl get <resource> -o yaml

6. Test connectivity
   kubectl exec <pod> -- <command>

7. Check related resources
   kubectl get all -l <label>

Lab 1: View Logs 

# Create Pod
kubectl run nginx --image=nginx

# View logs
kubectl logs nginx

# Follow logs
kubectl logs -f nginx

# Previous instance (if restarted)

Lab 2: Debug Crash Loop

Create failing Pod:

apiVersion: v1
kind: Pod
metadata:
  name: crash-pod
spec:
  containers:
  - name: app
    image: busybox
    command: ['sh', '-c', 'exit 1']

Debug:

kubectl apply -f pod.yaml
kubectl get pods
kubectl describe pod crash-pod
kubectl logs crash-pod
kubectl logs crash-pod --previous

Lab 3: Check Events 

# Create Pod
kubectl run test --image=nginx

# Watch events
kubectl get events --watch

# Get events for Pod
kubectl describe pod test | grep -A 10 Events

# All events sorted
kubectl get events --sort-by='.lastTimestamp'

Lab 4: Exec into Pod 

# Create Pod
kubectl run debug --image=nginx

# Execute command
kubectl exec debug -- ls /usr/share/nginx/html

# Interactive shell
kubectl exec -it debug -- /bin/bash

Lab 5: Port Forward 

# Create Pod
kubectl run nginx --image=nginx

# Port forward
kubectl port-forward nginx 8080:80

# In another terminal, test
curl http://localhost:8080
