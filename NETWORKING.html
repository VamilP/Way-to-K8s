ğŸ¯ Services: Stable Networking

The Problem

Pods are ephemeral:

  • Pod IPs change when Pods restart
  • How do clients find Pods?
  • How to load balance across Pods?

The Solution: Services

Service = Stable endpoint that:

  • Provides stable IP (ClusterIP)
  • Load balances to Pods
  • Uses labels/selectors to find Pods

ğŸ”§ Service Types

1. ClusterIP (Default)

Purpose: Internal cluster access only.

Features:

  • âœ… Stable IP within cluster
  • âœ… Load balancing
  • âŒ Not accessible from outside cluster

Example:

apiVersion: v1
kind: Service
metadata:
  name: nginx-service
spec:
  type: ClusterIP  # Default, can be omitted
  selector:
    app: nginx
  ports:
  - port: 80
    targetPort: 8080

Access:

# From within cluster
curl http://nginx-service.default.svc.cluster.local
# or
curl http://10.96.0.1

2. NodePort

Purpose: Expose Service on each Node's IP at a static port.

Features:

  • âœ… Accessible from outside cluster
  • âœ… Same port on all nodes
  • âš ï¸ Port range: 30000-32767

Example:

apiVersion: v1
kind: Service
metadata:
  name: nginx-service
spec:
  type: NodePort
  selector:
    app: nginx
  ports:
  - port: 80
    targetPort: 8080
    nodePort: 30080  # Optional, auto-assigned if omitted

Access:

http://<any-node-ip>:30080

Flow:

External Request â†’ Node IP:30080 â†’ kube-proxy â†’ Pods

3. LoadBalancer

Purpose: Expose Service externally using cloud provider's load balancer.

Features:

  • âœ… External IP provided by cloud
  • âœ… Cloud load balancer integration
  • âš ï¸ Cloud provider specific

Example:

apiVersion: v1
kind: Service
metadata:
  name: nginx-service
spec:
  type: LoadBalancer
  selector:
    app: nginx
  ports:
  - port: 80
    targetPort: 8080

Cloud Integration:

  • GKE: Creates GCP Load Balancer
  • EKS: Creates AWS ELB/ALB
  • AKS: Creates Azure Load Balancer

Access:

http://<external-ip>

4. ExternalName

Purpose: Map Service to external DNS name.

Example:

apiVersion: v1
kind: Service
metadata:
  name: external-db
spec:
  type: ExternalName
  externalName: database.example.com

Use Case: Access external services with Service abstraction.

ğŸ” Service Selectors & Endpoints

How Services Find Pods

Service uses selector to find Pods:

apiVersion: v1
kind: Service
metadata:
  name: nginx-service
spec:
  selector:
    app: nginx      # Matches Pods with label app=nginx
    env: production
  ports:
  - port: 80

Matching Pods:

metadata:
  labels:
    app: nginx
    env: production

Endpoints

Endpoints = List of Pod IPs that match Service selector.

Automatic Creation:

# Service created
kubectl create service clusterip nginx --tcp=80:8080

# Endpoints automatically created
kubectl get endpoints nginx
# Output:
# NAME    ENDPOINTS
# nginx   10.244.1.5:8080,10.244.1.6:8080,10.244.1.7:8080

Manual Endpoints (Headless Service):

apiVersion: v1
kind: Service
metadata:
  name: nginx-headless
spec:
  clusterIP: None  # Headless
  selector:
    app: nginx
  ports:
  - port: 80

Use Case: StatefulSets need stable DNS for each Pod.

ğŸŒ DNS & Service Discovery

CoreDNS

CoreDNS = DNS server for Kubernetes.

Default DNS:

<service-name>.<namespace>.svc.cluster.local

Examples:

nginx-service.default.svc.cluster.local
mysql.production.svc.cluster.local
api.staging.svc.cluster.local

Short Names:

  • Same namespace: nginx-service
  • Different namespace: nginx-service.default

DNS Records

Service DNS:

nginx-service.default.svc.cluster.local â†’ 10.96.0.1 (ClusterIP)

Pod DNS (Headless Service):

web-0.web.default.svc.cluster.local â†’ 10.244.1.5 (Pod IP)
web-1.web.default.svc.cluster.local â†’ 10.244.2.5 (Pod IP)

ğŸ”Œ Container Network Interface (CNI)

What is CNI?

CNI = Standard interface for networking plugins.

Purpose: Connect Pods to cluster network.

How CNI Works 

Pod created
    â†“
kubelet calls CNI plugin
    â†“
CNI plugin:
  1. Creates network namespace
  2. Creates virtual interface (veth pair)
  3. Connects to bridge/overlay
  4. Assigns IP address
  5. Sets up routes
    â†“
Pod has network connectivity

Popular CNI Plugins

  1. Calico

    • BGP-based routing
    • Network policies
    • Good for on-premises

  2. Flannel

    • Simple overlay network
    • VXLAN or host-gw
    • Easy to set up

  3. Weave Net

    • Overlay network
    • Automatic mesh
    • Encryption support

  4. Cilium

    • eBPF-based
    • High performance
    • Advanced networking

CNI Configuration

Location: /etc/cni/net.d/

Example (Calico):

{
  "name": "k8s-pod-network",
  "cniVersion": "0.3.1",
  "plugins": [
    {
      "type": "calico",
      "etcd_endpoints": "https://etcd:2379"
    }
  ]
}

ğŸšª Ingress: HTTP/HTTPS Routing

The Problem

Services expose TCP/UDP, not HTTP:

  • Need path-based routing (/api, /web)
  • Need host-based routing (api.example.com, web.example.com)
  • Need SSL/TLS termination

The Solution: Ingress

Ingress = HTTP/HTTPS routing rules.

Components:

  1. Ingress Resource (rules)
  2. Ingress Controller (implements rules)
