ğŸ§  Kubernetes API Overview

What is the Kubernetes API?

The Kubernetes API is a RESTful API that exposes:

  • Resources (Pods, Services, Deployments, etc.)
  • Operations (GET, POST, PUT, DELETE, PATCH, WATCH)
  • Grouped by API versions (v1, apps/v1, networking.k8s.io/v1)

API Structure 

https://<api-server>/api/v1/namespaces/<namespace>/pods/<pod-name>
https://<api-server>/apis/apps/v1/namespaces/<namespace>/deployments/<deployment-name>

API Groups:

  • Core API (/api/v1): Pods, Services, ConfigMaps, Secrets
  • Apps API (/apis/apps/v1): Deployments, ReplicaSets, StatefulSets
  • Networking API (/apis/networking.k8s.io/v1): Ingress, NetworkPolicy
  • Storage API (/apis/storage.k8s.io/v1): StorageClass, PersistentVolume

ğŸ“ Declarative vs Imperative

Imperative (How) 

# "Do this, then this, then this"
kubectl run nginx --image=nginx
kubectl scale deployment nginx --replicas=3
kubectl expose deployment nginx --port=80

Declarative (What) 

# "I want this state"
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx
spec:
  replicas: 3
  template:
    spec:
      containers:
      - name: nginx
        image: nginx

Kubernetes is Declarative:

  • You declare what you want
  • Kubernetes figures out how to achieve it
  • Kubernetes continuously reconciles to maintain desired state

ğŸ“„ YAML Structure Anatomy

Every Kubernetes resource has three main sections:

apiVersion: v1              # API version
kind: Pod                   # Resource type
metadata:                   # Identity and labels
  name: my-pod
  labels:
    app: nginx
spec:                       # Desired state
  containers:
  - name: nginx
    image: nginx:1.21
status:                     # Current state (read-only, managed by K8s)
  phase: Running
  ...

1. apiVersion

Purpose: Specifies which API version to use.

Common Versions:

  • v1 - Core API (Pods, Services, ConfigMaps, Secrets)
  • apps/v1 - Deployments, ReplicaSets, StatefulSets, DaemonSets
  • networking.k8s.io/v1 - Ingress, NetworkPolicy
  • storage.k8s.io/v1 - StorageClass, PersistentVolume
  • rbac.authorization.k8s.io/v1 - Roles, RoleBindings

How to find:

kubectl api-versions
kubectl explain pod.apiVersion

2. kind

Purpose: Specifies the resource type.

Common Kinds:

  • Pod
  • Deployment
  • Service
  • ConfigMap
  • Secret
  • Namespace
  • PersistentVolumeClaim

3. metadata

Purpose: Identity, labels, annotations, and ownership.

Key Fields:

metadata:
  name: my-pod              # Required: unique name in namespace
  namespace: default         # Optional: namespace (default: default)
  labels:                   # Key-value pairs for selection
    app: nginx
    env: production
  annotations:              # Non-identifying metadata
    description: "Web server"
  ownerReferences:          # Parent resource (managed by K8s)
    - apiVersion: apps/v1
      kind: ReplicaSet
      name: nginx-rs

4. spec

Purpose: Desired state (what you want).

Managed by: You (the user)

Example:

spec:
  replicas: 3               # Desired: 3 replicas
  containers:
  - name: nginx
    image: nginx:1.21
    ports:
    - containerPort: 80

5. status

Purpose: Current state (what actually exists).

Managed by: Kubernetes (read-only)

Example:

status:
  replicas: 3               # Current: 3 replicas
  readyReplicas: 3          # Ready: 3 replicas
  conditions:
  - type: Available
    status: "True"

ğŸ·ï¸ Labels & Selectors

Labels

Purpose: Key-value pairs attached to objects for identification and selection.

Rules:

  • Max 63 characters per key/value
  • Must start/end with alphanumeric
  • Can contain: -, _, .

Example:

metadata:
  labels:
    app: nginx
    env: production
    tier: frontend
    version: "1.21"

Selectors

Purpose: Select resources by labels.

Types:

  1. Equality-based (=, ==, !=)
  2. Set-based (in, notin, exists)

Example - Equality:

selector:
  matchLabels:
    app: nginx
    env: production

Example - Set-based:

selector:
  matchExpressions:
  - key: env
    operator: In
    values:
    - production
    - staging

ğŸ’» Essential kubectl Commands

Basic Commands 

# Get resources
kubectl get pods
kubectl get pods -n <namespace>
kubectl get all                    # All resources in namespace
kubectl get all --all-namespaces   # All resources in all namespaces

# Describe resource
kubectl describe pod <pod-name>
kubectl describe node <node-name>

# Create resource
kubectl create -f pod.yaml
kubectl apply -f pod.yaml          # Declarative (preferred)

# Delete resource
kubectl delete pod <pod-name>
kubectl delete -f pod.yaml

# Edit resource
kubectl edit pod <pod-name>

# Get resource YAML
kubectl get pod <pod-name> -o yaml
kubectl get pod <pod-name> -o json

Output Formats 

# YAML
kubectl get pod <pod-name> -o yaml

# JSON
kubectl get pod <pod-name> -o json

# Wide (more columns)
kubectl get pods -o wide

# Custom columns
kubectl get pods -o custom-columns=NAME:.metadata.name,STATUS:.status.phase

# JSONPath
kubectl get pods -o jsonpath='{.items[*].metadata.name}'

Filtering & Selection 

# By label
kubectl get pods -l app=nginx
kubectl get pods -l 'env in (production,staging)'

# By field
kubectl get pods --field-selector status.phase=Running

# By namespace
kubectl get pods -n kube-system

Debugging Commands 

# Logs
kubectl logs <pod-name>
kubectl logs <pod-name> -c <container-name>  # Multi-container pod
kubectl logs <pod-name> --previous           # Previous container instance

# Execute command in pod
kubectl exec <pod-name> -- ls
kubectl exec <pod-name> -it -- /bin/bash

# Port forward
kubectl port-forward <pod-name> 8080:80

# Top (resource usage)
kubectl top pods
kubectl top nodes

ğŸ§ª Hands-On Lab: Working with YAML

Lab 1: Create a Pod from YAML

Create pod.yaml:

apiVersion: v1
kind: Pod
metadata:
  name: nginx-pod
  labels:
    app: nginx
    env: dev
spec:
  containers:
  - name: nginx
    image: nginx:1.21
    ports:
    - containerPort: 80

Apply:

kubectl apply -f pod.yaml

# Verify
kubectl get pods
kubectl describe pod nginx-pod

Lab 2: View and Edit 

# Get YAML
kubectl get pod nginx-pod -o yaml > pod-full.yaml

# Edit live
kubectl edit pod nginx-pod

# Note: Some fields are immutable (like image in Pod)
# For changes, delete and recreate, or use Deployment

Lab 3: Labels and Selectors

Create multiple Pods:

kubectl run nginx-prod --image=nginx -l app=nginx,env=prod
kubectl run nginx-dev --image=nginx -l app=nginx,env=dev
kubectl run apache-prod --image=httpd -l app=apache,env=prod

Query by labels:

# All nginx pods
kubectl get pods -l app=nginx

# Production pods
kubectl get pods -l env=prod

# Nginx production pods
kubectl get pods -l app=nginx,env=prod

# Set-based selector
kubectl get pods -l 'env in (prod,staging)'

Lab 4: Understanding spec vs status 

# Create a Pod
kubectl apply -f pod.yaml

# Check spec (desired state)
kubectl get pod nginx-pod -o jsonpath='{.spec.containers[0].image}'
# Output: nginx:1.21

# Check status (current state)
kubectl get pod nginx-pod -o jsonpath='{.status.phase}'
# Output: Running

# Full status
kubectl get pod nginx-pod -o yaml | grep -A 20 status

ğŸ“ Key Takeaways

  1. Kubernetes API = RESTful API for cluster resources
  2. Declarative model = Declare what you want, K8s figures out how
  3. YAML structure = apiVersion, kind, metadata, spec, status
  4. metadata = Identity, labels, annotations
  5. spec = Desired state (you control)
  6. status = Current state (K8s manages, read-only)
  7. Labels = Key-value pairs for selection
  8. Selectors = Query resources by labels
  9. kubectl = CLI tool to interact with API
  10. kubeconfig = Configuration for kubectl
