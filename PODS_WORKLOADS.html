ğŸ“¦ What is a Pod?

The Fundamental Unit

Pod = The smallest deployable unit in Kubernetes.

Key Concept: A Pod contains one or more containers that:

  • Share the same network namespace (same IP)
  • Share the same storage volumes
  • Are scheduled together (all or nothing)
  • Run on the same node

Why Pods?

Docker runs containers. Kubernetes runs Pods.

Reason: Containers are too low-level. Pods provide:

  • Grouping: Related containers together
  • Shared networking: Containers can communicate via localhost
  • Shared storage: Containers can share volumes
  • Atomic unit: All containers start/stop together

Pod Analogy

Think of a Pod like a room in a hotel:

  • Containers = People in the room
  • Network = Room's phone (shared)
  • Storage = Room's safe (shared)
  • Scheduling = Room assignment (all or nothing)

Pod States

  1. Pending: Pod accepted, but containers not running

    • Waiting for scheduling
    • Pulling images
    • Waiting for volumes

  2. Running: Pod bound to node, all containers running

    • At least one container is running
    • Containers may be starting or restarting

  3. Succeeded: All containers terminated successfully

    • Exit code 0
    • Won't restart

  4. Failed: At least one container failed

    • Non-zero exit code
    • May restart (depending on restartPolicy)

  5. Unknown: Pod state couldn't be determined

    • Node communication lost

Container States (within Pod) 

status:
  containerStatuses:
  - name: nginx
    state:
      waiting:        # Container waiting to start
        reason: ImagePullBackOff
      running:        # Container running
        startedAt: "2024-01-01T10:00:00Z"
      terminated:     # Container stopped
        exitCode: 0
        finishedAt: "2024-01-01T11:00:00Z"

ğŸ” Init Containers

Purpose: Run containers that must complete before main containers start.

Use Cases:

  • Database migrations
  • Waiting for dependencies
  • Setting up configuration
  • Downloading data

Example:

apiVersion: v1
kind: Pod
metadata:
  name: app-pod
spec:
  initContainers:
  - name: wait-db
    image: busybox
    command: ['sh', '-c', 'until nc -z db 5432; do sleep 1; done']
  containers:
  - name: app
    image: my-app:1.0

Execution Flow:

3. StatefulSet (Stateful Apps)

Purpose: Manage stateful applications with stable identity.

Features:

  • âœ… Stable network identity (hostname)
  • âœ… Stable storage (persistent volumes)
  • âœ… Ordered deployment/scaling
  • âœ… Ordered rolling updates

When to Use:

  • Databases (MySQL, PostgreSQL, MongoDB)
  • Message queues (RabbitMQ, Kafka)
  • Any app needing stable identity/storage

Example:

apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: mysql
spec:
  serviceName: mysql
  replicas: 3
  selector:
    matchLabels:
      app: mysql
  template:
    metadata:
      labels:
        app: mysql
    spec:
      containers:
      - name: mysql
        image: mysql:8.0
        volumeMounts:
        - name: data
          mountPath: /var/lib/mysql
  volumeClaimTemplates:
  - metadata:
      name: data
    spec:
      accessModes: ["ReadWriteOnce"]
      resources:
        requests:
          storage: 10Gi

Pod Naming:

Stable Identity:

  • Hostname: mysql-0.mysql.default.svc.cluster.local
  • Storage: Each Pod gets its own PersistentVolume

5. Job (One-Time Tasks)

Purpose: Run a Pod until completion.

Features:

  • âœ… Runs until successful completion
  • âœ… Retries on failure
  • âœ… Parallel execution support

When to Use:

  • Database migrations
  • Data processing
  • Batch jobs
  • One-time tasks

Example:

apiVersion: batch/v1
kind: Job
metadata:
  name: migrate-db
spec:
  template:
    spec:
      containers:
      - name: migrate
        image: migrate:1.0
        command: ['sh', '-c', 'python migrate.py']
      restartPolicy: Never
  backoffLimit: 3  # Retry 3 times on failure

6. CronJob (Scheduled Tasks)

Purpose: Run Jobs on a schedule.

Features:

  • âœ… Cron-like scheduling
  • âœ… Creates Jobs on schedule
  • âœ… Timezone support

When to Use:

  • Scheduled backups
  • Periodic reports
  • Cleanup tasks
  • Data synchronization

Example:

apiVersion: batch/v1
kind: CronJob
metadata:
  name: backup-db
spec:
  schedule: "0 2 * * *"  # Every day at 2 AM
  jobTemplate:
    spec:
      template:
        spec:
          containers:
          - name: backup
            image: backup:1.0
            command: ['sh', '-c', 'backup.sh']
          restartPolicy: OnFailure

ğŸ” Health Checks: Probes

Liveness Probe

Purpose: Know when to restart a container.

Example:

containers:
- name: nginx
  image: nginx:1.21
  livenessProbe:
    httpGet:
      path: /health
      port: 80
    initialDelaySeconds: 30
    periodSeconds: 10

Flow:

Container running â†’ Liveness probe fails â†’ Container restarted

Readiness Probe

Purpose: Know when a container is ready to receive traffic.

Example:

containers:
- name: nginx
  image: nginx:1.21
  readinessProbe:
    httpGet:
      path: /ready
      port: 80
    initialDelaySeconds: 5
    periodSeconds: 5

Flow:

Container starting â†’ Readiness probe passes â†’ Pod receives traffic

Startup Probe

Purpose: Know when a container has started (for slow-starting apps).

Example:

containers:
- name: app
  image: my-app:1.0
  startupProbe:
    httpGet:
      path: /startup
      port: 8080
    failureThreshold: 30
    periodSeconds: 10

Lab 2: Create a Deployment

Create deployment.yaml:

apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.21
        ports:
        - containerPort: 80

Apply:

kubectl apply -f deployment.yaml

# Check Deployment
kubectl get deployment
kubectl get replicaset
kubectl get pods

# Scale Deployment
kubectl scale deployment nginx-deployment --replicas=5

# Update image
kubectl set image deployment/nginx-deployment nginx=nginx:1.22

# Watch rolling update
kubectl rollout status deployment/nginx-deployment

# Rollback
kubectl rollout undo deployment/nginx-deployment

ğŸ“ Key Takeaways

  1. Pod = Smallest deployable unit (1+ containers)
  2. Deployment = Stateless apps with rolling updates
  3. StatefulSet = Stateful apps with stable identity/storage
  4. DaemonSet = One Pod per node
  5. Job = One-time tasks
  6. CronJob = Scheduled tasks
  7. Probes = Health checks (liveness, readiness, startup)
  8. Init Containers = Run before main containers
  9. Pod States = Pending, Running, Succeeded, Failed, Unknown
  10. Choose workload based on app requirements
