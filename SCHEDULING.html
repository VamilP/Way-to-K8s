ğŸ’¾ Resource Requests & Limits

Why Resource Management?

Problem: Containers can consume all node resources, starving other Pods.

Solution: Resource requests and limits.

Requests vs Limits

Request = Guaranteed minimum resources.

Limit = Maximum resources allowed.

Example:

apiVersion: v1
kind: Pod
metadata:
  name: app-pod
spec:
  containers:
  - name: app
    image: my-app:1.0
    resources:
      requests:
        memory: "64Mi"
        cpu: "250m"      # 0.25 CPU
      limits:
        memory: "128Mi"
        cpu: "500m"      # 0.5 CPU

CPU Units

  • 1 = 1 CPU core
  • 1000m = 1 CPU core
  • 500m = 0.5 CPU core
  • 250m = 0.25 CPU core

Memory Units

  • 1Gi = 1024 MiB
  • 1G = 1000 MB (decimal)
  • 128Mi = 128 MiB
  • 512M = 512 MB (decimal)

What Happens When Limits Are Exceeded?

CPU:

  • Throttled (slowed down)
  • Pod continues running

Memory:

  • OOMKilled (Out Of Memory)
  • Pod terminated and restarted

ğŸ·ï¸ Quality of Service (QoS) Classes

Kubernetes assigns QoS classes based on resource configuration.

1. Guaranteed

Requirements:

  • All containers have requests = limits
  • Requests and limits set for both CPU and memory

Example:

resources:
  requests:
    memory: "64Mi"
    cpu: "250m"
  limits:
    memory: "64Mi"
    cpu: "250m"

Priority: Highest (last to be evicted)

2. Burstable

Requirements:

  • At least one container has requests < limits
  • Or requests set but limits not set

Example:

resources:
  requests:
    memory: "64Mi"
    cpu: "250m"
  limits:
    memory: "128Mi"
    cpu: "500m"

Priority: Medium

3. BestEffort

Requirements:

  • No requests or limits set

Example:

# No resources section

Priority: Lowest (first to be evicted)

QoS Eviction Order 

Node runs out of resources
    â†“
1. BestEffort Pods evicted first
    â†“
2. Burstable Pods evicted (if needed)
    â†“
3. Guaranteed Pods evicted last (only if critical)

ğŸ”— Node Affinity

What is Affinity?

Affinity = Advanced scheduling rules (preferred vs required).

Types of Affinity

  1. Required (requiredDuringSchedulingIgnoredDuringExecution)

    • Must match (hard requirement)

  2. Preferred (preferredDuringSchedulingIgnoredDuringExecution)

    • Should match (soft preference)

Node Affinity Example 

apiVersion: v1
kind: Pod
metadata:
  name: app-pod
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: disktype
            operator: In
            values:
            - ssd
            - nvme
      preferredDuringSchedulingIgnoredDuringExecution:
      - weight: 100
        preference:
          matchExpressions:
          - key: zone
            operator: In
            values:
            - us-west-1
  containers:
  - name: app
    image: my-app:1.0

Operators:

  • In - Value in list
  • NotIn - Value not in list
  • Exists - Key exists
  • DoesNotExist - Key doesn't exist
  • Gt - Greater than (for numeric)
  • Lt - Less than (for numeric)

ğŸ”„ Pod Affinity & Anti-Affinity

Pod Affinity

Purpose: Schedule Pods near other Pods.

Use Case: Co-locate related Pods for performance.

Example:

apiVersion: v1
kind: Pod
metadata:
  name: app-pod
spec:
  affinity:
    podAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
      - labelSelector:
          matchExpressions:
          - key: app
            operator: In
            values:
            - database
        topologyKey: zone
  containers:
  - name: app
    image: my-app:1.0

Meaning: Schedule on same zone as database Pods.

Pod Anti-Affinity

Purpose: Schedule Pods away from other Pods.

Use Case: High availability (spread across nodes/zones).

Example:

apiVersion: apps/v1
kind: Deployment
metadata:
  name: web
spec:
  replicas: 3
  template:
    spec:
      affinity:
        podAntiAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
          - labelSelector:
              matchExpressions:
              - key: app
                operator: In
                values:
                - web
            topologyKey: kubernetes.io/hostname
      containers:
      - name: web
        image: nginx

Meaning: Don't schedule multiple web Pods on same node.

ğŸš« Taints & Tolerations

What are Taints?

Taint = Mark on node that repels Pods (unless tolerated).

Use Cases:

  • Dedicated nodes (GPU, special hardware)
  • Node maintenance
  • Preventing Pods from scheduling

Taint Format 

key=value:effect

Effects:

  • NoSchedule - Don't schedule new Pods
  • PreferNoSchedule - Prefer not to schedule
  • NoExecute - Evict existing Pods

Tainting a Node 

# Taint node
kubectl taint nodes node-1 key=value:NoSchedule

# Remove taint
kubectl taint nodes node-1 key=value:NoSchedule-

Tolerations

Toleration = Allows Pod to be scheduled on tainted node.

Example:

apiVersion: v1
kind: Pod
metadata:
  name: app-pod
spec:
  tolerations:
  - key: "key"
    operator: "Equal"
    value: "value"
    effect: "NoSchedule"
  containers:
  - name: app
    image: my-app:1.0

Operators:

  • Equal - Exact match
  • Exists - Key exists (ignore value)

Common Use Case: Dedicated GPU Nodes 

# Taint GPU nodes
kubectl taint nodes gpu-node-1 nvidia.com/gpu=true:NoSchedule

# Pod with GPU toleration
apiVersion: v1
kind: Pod
metadata:
  name: gpu-pod
spec:
  tolerations:
  - key: "nvidia.com/gpu"
    operator: "Exists"
    effect: "NoSchedule"
  containers:
  - name: gpu-app
    image: gpu-app:1.0

ğŸ›¡ï¸ Pod Disruption Budgets (PDB)

What is a PDB?

PDB = Limits number of Pods that can be disrupted simultaneously.

Use Cases:

  • Ensure minimum availability during updates
  • Prevent too many Pods from being evicted

PDB Example 

apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: web-pdb
spec:
  minAvailable: 2
  selector:
    matchLabels:
      app: web

Alternatives:

  • minAvailable - Minimum Pods available
  • maxUnavailable - Maximum Pods unavailable

Example with maxUnavailable:

apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: web-pdb
spec:
  maxUnavailable: 1
  selector:
    matchLabels:
      app: web
