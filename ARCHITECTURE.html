ğŸ›ï¸ Control Plane Components

The Control Plane is the "brain" of Kubernetes. It makes all decisions and maintains cluster state.

1. kube-apiserver

Role: The front-end API server for Kubernetes.

Responsibilities:

  • Exposes Kubernetes API (REST API)
  • Validates and processes API requests
  • Authenticates and authorizes requests
  • Writes to etcd
  • Serves as the only component that talks to etcd

Key Points:

  • Single source of truth for API
  • Stateless (can run multiple instances)
  • All components communicate through API server

2. etcd

Role: Distributed key-value store (Kubernetes' database).

Responsibilities:

  • Stores all cluster state
  • Stores configuration data
  • Stores secrets (encrypted at rest)
  • Provides watch API for changes

Key Points:

  • Only kube-apiserver writes to etcd
  • Highly available (typically 3 or 5 nodes)
  • All cluster data lives here

What's Stored:

  • Pod definitions
  • Service definitions
  • ConfigMaps
  • Secrets
  • Node information
  • Cluster state

âš ï¸ Important: etcd is critical. If it's lost, cluster state is lost!

3. kube-scheduler

Role: Assigns Pods to Nodes.

Responsibilities:

  • Watches for newly created Pods (no assigned node)
  • Selects a node for each Pod
  • Considers resource requirements, constraints, affinity

Scheduling Process:

1. Filter nodes (which nodes can run this Pod?)
   - Resource availability (CPU, memory)
   - Node selectors
   - Taints/tolerations
   - Affinity rules

2. Score nodes (which node is best?)
   - Resource balance
   - Affinity preferences
   - Anti-affinity preferences

3. Bind Pod to selected node

Example:

Pod created â†’ Scheduler evaluates nodes â†’ Assigns to Node-2

4. kube-controller-manager

Role: Runs controller processes (control loops).

Responsibilities:

  • Maintains desired state
  • Reconciles current state with desired state
  • Runs multiple controllers

Controllers Include:

  • Deployment Controller: Maintains desired replicas
  • ReplicaSet Controller: Ensures correct number of Pods
  • Node Controller: Monitors node health
  • Job Controller: Manages Jobs
  • Namespace Controller: Manages namespaces
  • ServiceAccount Controller: Manages service accounts

Control Loop Pattern:

1. Observe current state
2. Compare with desired state
3. Take action to reconcile
4. Repeat

Example - Deployment Controller:

Desired: 3 replicas
Current: 2 replicas
Action: Create 1 more Pod

5. cloud-controller-manager

Role: Integrates with cloud provider APIs.

Responsibilities:

  • Node controller (cloud-specific)
  • Route controller (cloud load balancers)
  • Service controller (cloud load balancers)
  • Volume controller (cloud storage)

Key Point: Only runs if using a cloud provider (GCP, AWS, Azure).

ğŸ”§ Worker Node Components

Worker Nodes are the "muscles" that run your workloads.

1. kubelet

Role: Node agent that communicates with Control Plane.

Responsibilities:

  • Registers node with cluster
  • Watches for Pod assignments
  • Starts/stops containers via Container Runtime
  • Reports Pod and node status to API server
  • Executes liveness/readiness probes
  • Mounts volumes

Key Points:

  • Runs on every node
  • Primary interface between node and cluster
  • Ensures containers are running as specified

kubelet Workflow:

1. kubelet receives Pod spec from API server
2. Pulls container image
3. Creates Pod (network namespace, volumes)
4. Starts containers via Container Runtime
5. Monitors Pod health
6. Reports status back to API server

2. kube-proxy

Role: Network proxy and load balancer.

Responsibilities:

  • Maintains network rules on nodes
  • Enables Service abstraction (ClusterIP, NodePort, LoadBalancer)
  • Implements load balancing
  • Uses iptables or IPVS

How It Works:

Service created â†’ kube-proxy creates iptables rules â†’ Traffic routed to Pods

Example:

Service "my-app" â†’ ClusterIP: 10.96.0.1
kube-proxy creates rule:
  Traffic to 10.96.0.1 â†’ Forward to Pod IPs (10.244.1.2, 10.244.1.3)

3. Container Runtime

Role: Runs containers.

Common Runtimes:

  • containerd (default, recommended)
  • CRI-O
  • Docker (deprecated, uses containerd)

Responsibilities:

  • Pulling container images
  • Starting/stopping containers
  • Managing container lifecycle

Interface: Uses CRI (Container Runtime Interface)

ğŸ› Troubleshooting Component Issues

API Server Issues

Symptoms:

  • kubectl commands hang or fail
  • "connection refused" errors

Debug:

# Check API server status
kubectl get cs

# Check API server logs (on control plane node)
kubectl logs -n kube-system kube-apiserver-<node-name>

etcd Issues

Symptoms:

  • Cluster state not updating
  • "etcd cluster is unavailable"

Debug:

# Check etcd pods
kubectl get pods -n kube-system | grep etcd

# Check etcd health (if accessible)
etcdctl endpoint health

Scheduler Issues

Symptoms:

  • Pods stuck in "Pending" state
  • No scheduling events

Debug:

# Check scheduler pod
kubectl get pods -n kube-system | grep scheduler

# Check scheduler logs
kubectl logs -n kube-system kube-scheduler-<node-name>

# Check why Pod isn't scheduled
kubectl describe pod <pod-name> | grep -A 10 Events

kubelet Issues

Symptoms:

  • Pods not starting on node
  • Node shows "NotReady"

Debug:

# Check node status
kubectl describe node <node-name>

# Check kubelet logs (on node)
journalctl -u kubelet -f

ğŸ“ Key Takeaways

  1. Control Plane = Brain (makes decisions)
  2. Worker Nodes = Muscles (execute workloads)
  3. kube-apiserver = Front door (all requests go through it)
  4. etcd = Database (stores all state)
  5. kube-scheduler = Assigns Pods to nodes
  6. kube-controller-manager = Maintains desired state
  7. kubelet = Node agent (runs Pods)
  8. kube-proxy = Network proxy (Services)
  9. Control loops = Continuous reconciliation
  10. HA = Multiple Control Plane nodes for production
