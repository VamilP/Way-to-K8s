ğŸš¨ The Configuration Problem

Challenges

  1. Configuration changes require rebuilding images
  2. Environment-specific configs (dev, staging, prod)
  3. Sensitive data (passwords, API keys, certificates)
  4. Config updates without Pod restarts (sometimes)

Kubernetes Solution

  • ConfigMaps = Non-sensitive configuration
  • Secrets = Sensitive data (passwords, tokens, keys)

ğŸ“‹ ConfigMaps

What is a ConfigMap?

ConfigMap = API object for storing non-sensitive configuration data.

Use Cases:

  • Configuration files
  • Environment variables
  • Command-line arguments
  • Any non-sensitive data

Creating ConfigMaps

Method 1: From Literal Values 

kubectl create configmap my-config \
  --from-literal=key1=value1 \
  --from-literal=key2=value2

Method 2: From File 

# Create config file
echo "server.port=8080" > app.properties

# Create ConfigMap
kubectl create configmap app-config --from-file=app.properties

Method 3: From YAML 

apiVersion: v1
kind: ConfigMap
metadata:
  name: my-config
data:
  key1: value1
  key2: value2
  app.properties: |
    server.port=8080
    server.host=0.0.0.0

Using ConfigMaps in Pods

Method 1: Environment Variables 

apiVersion: v1
kind: Pod
metadata:
  name: app-pod
spec:
  containers:
  - name: app
    image: my-app:1.0
    env:
    - name: DATABASE_HOST
      valueFrom:
        configMapKeyRef:
          name: my-config
          key: db_host
    - name: DATABASE_PORT
      valueFrom:
        configMapKeyRef:
          name: my-config
          key: db_port

Method 2: All Keys as Environment Variables 

apiVersion: v1
kind: Pod
metadata:
  name: app-pod
spec:
  containers:
  - name: app
    image: my-app:1.0
    envFrom:
    - configMapRef:
        name: my-config

Method 3: Volume Mount (Files) 

apiVersion: v1
kind: Pod
metadata:
  name: app-pod
spec:
  containers:
  - name: app
    image: my-app:1.0
    volumeMounts:
    - name: config
      mountPath: /etc/config
  volumes:
  - name: config
    configMap:
      name: my-config

Result: ConfigMap keys become files in /etc/config/

Method 4: Specific Key as File 

volumes:
- name: config
  configMap:
    name: my-config
    items:
    - key: app.properties
      path: app.properties

ğŸ” Secrets

What is a Secret?

Secret = API object for storing sensitive data.

Use Cases:

  • Passwords
  • API keys
  • TLS certificates
  • OAuth tokens
  • Database credentials

âš ï¸ Important: Secrets Are NOT Encrypted by Default

Reality:

  • Secrets are base64 encoded (not encrypted)
  • Anyone with API access can read secrets
  • Secrets are stored in etcd (encrypted at rest if configured)
  • RBAC controls access

Best Practices:

  • Enable etcd encryption at rest
  • Use RBAC to limit access
  • Consider external secret managers (Vault, AWS Secrets Manager)

Creating Secrets

Method 1: From Literal Values 

kubectl create secret generic my-secret \
  --from-literal=username=admin \
  --from-literal=password=secret123

Method 2: From File 

# Create files
echo -n 'admin' > username.txt
echo -n 'secret123' > password.txt

# Create Secret
kubectl create secret generic my-secret \
  --from-file=username=username.txt \
  --from-file=password=password.txt

Method 3: From YAML (Base64 Encoded) 

apiVersion: v1
kind: Secret
metadata:
  name: my-secret
type: Opaque
data:
  username: YWRtaW4=  # base64('admin')
  password: c2VjcmV0MTIz  # base64('secret123')

Encoding:

echo -n 'admin' | base64
# Output: YWRtaW4=

Secret Types

Common Types:

  • Opaque - Arbitrary user-defined data
  • kubernetes.io/dockerconfigjson - Docker registry credentials
  • kubernetes.io/tls - TLS certificate and key
  • kubernetes.io/service-account-token - Service account token

Docker Registry Secret 

kubectl create secret docker-registry regcred \
  --docker-server=docker.io \
  --docker-username=myuser \
  --docker-password=mypassword \
  --docker-email=myemail@example.com

TLS Secret 

kubectl create secret tls tls-secret \
  --cert=tls.crt \
  --key=tls.key

Using Secrets in Pods

Method 1: Environment Variables 

apiVersion: v1
kind: Pod
metadata:
  name: app-pod
spec:
  containers:
  - name: app
    image: my-app:1.0
    env:
    - name: DB_USERNAME
      valueFrom:
        secretKeyRef:
          name: my-secret
          key: username
    - name: DB_PASSWORD
      valueFrom:
        secretKeyRef:
          name: my-secret
          key: password

Method 2: All Keys as Environment Variables 

apiVersion: v1
kind: Pod
metadata:
  name: app-pod
spec:
  containers:
  - name: app
    image: my-app:1.0
    envFrom:
    - secretRef:
        name: my-secret

Method 3: Volume Mount (Files) 

apiVersion: v1
kind: Pod
metadata:
  name: app-pod
spec:
  containers:
  - name: app
    image: my-app:1.0
    volumeMounts:
    - name: secret
      mountPath: /etc/secret
      readOnly: true
  volumes:
  - name: secret
    secret:
      secretName: my-secret

Result: Secret keys become files in /etc/secret/

Method 4: Specific Key as File 

volumes:
- name: secret
  secret:
    secretName: my-secret
    items:
    - key: username
      path: db-username
    - key: password
      path: db-password

ğŸ” Security Best Practices

1. Enable etcd Encryption at Rest

Why: Secrets stored in etcd should be encrypted.

How: Configure encryption provider in kube-apiserver.

2. Use RBAC

Limit access to Secrets:

apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: secret-reader
rules:
- apiGroups: [""]
  resources: ["secrets"]
  resourceNames: ["my-secret"]
  verbs: ["get"]

3. External Secret Managers

For production:

  • HashiCorp Vault
  • AWS Secrets Manager
  • Azure Key Vault
  • Google Secret Manager

Use: External Secrets Operator to sync secrets.

4. Rotate Secrets Regularly

Best Practice: Rotate secrets periodically.

Tools: External secret managers with rotation policies.

5. Don't Commit Secrets to Git

Never commit:

  • Secret YAML files with real values
  • Base64-encoded secrets (easily decoded)

Use:

  • Secret management tools
  • CI/CD secret injection
  • Git-secrets scanning

Lab 1: Create and Use ConfigMap

Create ConfigMap:

kubectl create configmap app-config \
  --from-literal=server.port=8080 \
  --from-literal=server.host=0.0.0.0

# Verify
kubectl get configmap app-config
kubectl describe configmap app-config

Use in Pod:

apiVersion: v1
kind: Pod
metadata:
  name: app-pod
spec:
  containers:
  - name: app
    image: nginx
    env:
    - name: SERVER_PORT
      valueFrom:
        configMapKeyRef:
          name: app-config
          key: server.port

Apply and verify:

kubectl apply -f pod.yaml
kubectl exec app-pod -- env | grep SERVER_PORT

Lab 2: ConfigMap as Volume

Create ConfigMap with file:

apiVersion: v1
kind: ConfigMap
metadata:
  name: nginx-config
data:
  nginx.conf: |
    server {
        listen 80;
        server_name localhost;
        location / {
            root /usr/share/nginx/html;
        }
    }

Use in Pod:

apiVersion: v1
kind: Pod
metadata:
  name: nginx-pod
spec:
  containers:
  - name: nginx
    image: nginx
    volumeMounts:
    - name: config
      mountPath: /etc/nginx/conf.d
  volumes:
  - name: config
    configMap:
      name: nginx-config

Verify:

kubectl apply -f configmap.yaml
kubectl apply -f pod.yaml
kubectl exec nginx-pod -- cat /etc/nginx/conf.d/nginx.conf

ğŸ› Troubleshooting

ConfigMap/Secret Not Found 

# Check if exists
kubectl get configmap <name>
kubectl get secret <name>

# Check Pod events
kubectl describe pod <pod-name>

# Common issue: Wrong namespace
kubectl get configmap <name> -n <namespace>

Environment Variable Not Set 

# Check Pod spec
kubectl get pod <pod-name> -o yaml | grep -A 10 env

# Check actual environment
kubectl exec <pod-name> -- env

# Check ConfigMap/Secret keys
kubectl get configmap <name> -o yaml
kubectl get secret <name> -o yaml

File Not Mounted 

# Check volume mounts
kubectl describe pod <pod-name> | grep -A 10 Mounts

# Check if files exist
kubectl exec <pod-name> -- ls -la /path/to/mount

ğŸ“ Key Takeaways

  1. ConfigMaps = Non-sensitive configuration
  2. Secrets = Sensitive data (base64 encoded, not encrypted)
  3. Injection methods = Environment variables, volume mounts
  4. Secrets are NOT encrypted by default = Enable etcd encryption
  5. Use RBAC = Limit access to secrets
  6. External secret managers = Production best practice
  7. Don't commit secrets = Use secret management tools
  8. Rotate secrets = Security best practice
  9. ConfigMaps/Secrets are namespaced = Scope to namespace
  10. Volume mounts = Files, Environment variables = env vars
