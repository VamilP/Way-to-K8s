üîê Authentication vs Authorization

Authentication (Who are you?)

Purpose: Verify identity.

Methods:

  • Certificates
  • Tokens
  • Service Accounts
  • OpenID Connect (OIDC)

Authorization (What can you do?)

Purpose: Verify permissions.

Methods:

  • RBAC (Role-Based Access Control)
  • ABAC (Attribute-Based Access Control)
  • Node Authorization
  • Webhook Authorization

üë§ Service Accounts

What is a Service Account?

Service Account = Identity for Pods to authenticate to API server.

Default: Every namespace has default Service Account.

Creating Service Accounts 

apiVersion: v1
kind: ServiceAccount
metadata:
  name: my-serviceaccount
  namespace: default

Or via kubectl:

kubectl create serviceaccount my-sa

Using Service Account in Pod 

apiVersion: v1
kind: Pod
metadata:
  name: app-pod
spec:
  serviceAccountName: my-serviceaccount
  containers:
  - name: app
    image: my-app:1.0

Default Behavior:

  • If not specified, uses default Service Account
  • Service Account token mounted at /var/run/secrets/kubernetes.io/serviceaccount

üé≠ RBAC: Roles & ClusterRoles

What is RBAC?

RBAC = Role-Based Access Control.

Components:

  • Role / ClusterRole = Set of permissions
  • RoleBinding / ClusterRoleBinding = Grants role to user/group/SA

Role vs ClusterRole

Feature Role ClusterRole
Scope Namespace Cluster-wide
Use Case Namespace-specific permissions Cluster-wide permissions

Role Example 

apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: default
  name: pod-reader
rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "watch", "list"]

Verbs:

  • get, list, watch - Read
  • create, update, patch - Write
  • delete - Delete
  • * - All verbs

Resources:

  • pods, services, deployments, secrets, configmaps, etc.

ClusterRole Example 

apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: cluster-admin
rules:
- apiGroups: ["*"]
  resources: ["*"]
  verbs: ["*"]

üîó RoleBinding & ClusterRoleBinding

RoleBinding

Purpose: Grant Role to user/group/ServiceAccount in namespace.

Example:

apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: read-pods
  namespace: default
subjects:
- kind: User
  name: alice
  apiGroup: rbac.authorization.k8s.io
- kind: ServiceAccount
  name: my-sa
  namespace: default
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io

ClusterRoleBinding

Purpose: Grant ClusterRole to user/group/ServiceAccount cluster-wide.

Example:

apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: cluster-admin-binding
subjects:
- kind: User
  name: admin
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: ClusterRole
  name: cluster-admin
  apiGroup: rbac.authorization.k8s.io

üõ°Ô∏è Pod Security Standards

What are Pod Security Standards?

PSS = Policies that enforce security best practices for Pods.

Levels:

  1. Privileged - No restrictions
  2. Baseline - Minimal restrictions
  3. Restricted - Maximum restrictions

Enforcing Pod Security

Via Namespace Labels:

apiVersion: v1
kind: Namespace
metadata:
  name: production
  labels:
    pod-security.kubernetes.io/enforce: restricted
    pod-security.kubernetes.io/audit: restricted
    pod-security.kubernetes.io/warn: restricted

Via Admission Controller:

  • Pod Security Admission (PSA) - Built-in
  • OPA Gatekeeper - External

Security Context

Pod Security Context:

apiVersion: v1
kind: Pod
metadata:
  name: secure-pod
spec:
  securityContext:
    runAsNonRoot: true
    runAsUser: 1000
    fsGroup: 2000
  containers:
  - name: app
    image: my-app:1.0
    securityContext:
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem: true
      capabilities:
        drop:
        - ALL
        add:
        - NET_BIND_SERVICE

üåê Network Policies

What is a Network Policy?

Network Policy = Firewall rules for Pods.

Purpose: Control traffic between Pods.

Note: Requires CNI plugin with NetworkPolicy support (Calico, Cilium, etc.)

Network Policy Example

Deny All (Default):

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
  namespace: default
spec:
  podSelector: {}
  policyTypes:
  - Ingress
  - Egress

Allow Ingress from Specific Pods:

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend
  namespace: default
spec:
  podSelector:
    matchLabels:
      app: backend
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: frontend
    ports:
    - protocol: TCP
      port: 8080

Allow Egress to Specific Services:

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-dns
  namespace: default
spec:
  podSelector: {}
  policyTypes:
  - Egress
  egress:
  - to:
    - namespaceSelector:
        matchLabels:
          name: kube-system
    ports:
    - protocol: UDP
      port: 53

üîê Security Best Practices

1. Enable RBAC

Always enabled by default in modern Kubernetes.

Verify:

kubectl api-versions | grep rbac

2. Use Least Privilege

Principle: Grant minimum permissions needed.

Example:

  • Don't grant cluster-admin to developers
  • Create specific Roles for specific tasks

3. Rotate Certificates

Kubernetes certificates expire:

  • Default: 1 year
  • Rotate before expiration

Check expiration:

kubeadm certs check-expiration

4. Enable etcd Encryption

Encrypt secrets at rest:

  • Configure encryption provider
  • Restart API server

5. Use Network Policies

Isolate Pods:

  • Default deny
  • Explicit allow rules
  • Test thoroughly

6. Scan Images

Use image scanners:

  • Trivy
  • Clair
  • Snyk

7. Keep Kubernetes Updated

Security patches:

  • Update control plane
  • Update worker nodes
  • Update kubectl

üêõ Troubleshooting RBAC

Permission Denied 

# Check user/Service Account
kubectl auth can-i get pods --as=system:serviceaccount:default:my-sa

# Check bindings
kubectl get rolebindings,clusterrolebindings --all-namespaces

# Check roles
kubectl get roles,clusterroles

Service Account Not Working 

# Check Service Account exists
kubectl get serviceaccount

# Check Pod uses correct Service Account
kubectl describe pod <pod-name> | grep Service

# Check token mounted
kubectl exec <pod-name> -- ls /var/run/secrets/kubernetes.io/serviceaccount

üéì Key Takeaways

  1. Authentication = Who are you?
  2. Authorization = What can you do?
  3. Service Accounts = Identity for Pods
  4. Roles = Namespace-scoped permissions
  5. ClusterRoles = Cluster-scoped permissions
  6. RoleBindings = Grant Roles to subjects
  7. Pod Security Standards = Enforce security policies
  8. Network Policies = Firewall for Pods
  9. Least Privilege = Grant minimum permissions
  10. RBAC = Primary authorization method
