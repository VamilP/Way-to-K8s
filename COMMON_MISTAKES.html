üåê Networking Mistakes

Mistake 1: Using NodePort for Production

‚ùå Wrong:

apiVersion: v1
kind: Service
metadata:
  name: my-app
spec:
  type: NodePort
  ports:
  - port: 80
    nodePort: 30080

Why it's wrong:

  • Exposes service on every node
  • Requires firewall rules for each node
  • Not suitable for production

‚úÖ Correct:

apiVersion: v1
kind: Service
metadata:
  name: my-app
spec:
  type: ClusterIP  # Or LoadBalancer
  ports:
  - port: 80
---
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-app-ingress
spec:
  rules:
  - host: myapp.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: my-app
            port:
              number: 80

Mistake 2: Misusing LoadBalancer

‚ùå Wrong:

# Creating LoadBalancer for every service
apiVersion: v1
kind: Service
metadata:
  name: internal-api
spec:
  type: LoadBalancer  # Expensive!

Why it's wrong:

  • Each LoadBalancer costs money (cloud provider)
  • Use only for external-facing services

‚úÖ Correct:

# Internal service
apiVersion: v1
kind: Service
metadata:
  name: internal-api
spec:
  type: ClusterIP  # Internal only

# External service
apiVersion: v1
kind: Service
metadata:
  name: public-api
spec:
  type: LoadBalancer  # Only for external

Mistake 3: Hardcoding IP Addresses

‚ùå Wrong:

env:
- name: DB_HOST
  value: "10.244.1.5"  # Pod IP changes!

‚úÖ Correct:

env:
- name: DB_HOST
  value: "mysql"  # Use Service name

üíæ Resource Management Mistakes

Mistake 4: Not Setting Resource Limits

‚ùå Wrong:

containers:
- name: app
  image: my-app:1.0
  # No resources specified

Why it's wrong:

  • Pods can consume all node resources
  • Other Pods starved
  • Node instability

‚úÖ Correct:

containers:
- name: app
  image: my-app:1.0
  resources:
    requests:
      memory: "128Mi"
      cpu: "100m"
    limits:
      memory: "256Mi"
      cpu: "200m"

Mistake 5: Setting Limits Too High

‚ùå Wrong:

resources:
  requests:
    memory: "10Gi"
    cpu: "4"
  limits:
    memory: "20Gi"
    cpu: "8"

Why it's wrong:

  • Wastes resources
  • Prevents other Pods from scheduling
  • Over-provisioning

‚úÖ Correct:

resources:
  requests:
    memory: "256Mi"
    cpu: "250m"
  limits:
    memory: "512Mi"
    cpu: "500m"

Mistake 6: Requests = Limits (Always)

‚ùå Wrong:

# Always setting requests = limits
resources:
  requests:
    memory: "256Mi"
    cpu: "250m"
  limits:
    memory: "256Mi"  # Same as requests
    cpu: "250m"      # Same as requests

Why it's wrong:

  • No room for burst traffic
  • All Pods become Guaranteed QoS
  • Less flexibility

‚úÖ Correct:

# Allow some headroom
resources:
  requests:
    memory: "256Mi"
    cpu: "250m"
  limits:
    memory: "512Mi"  # 2x requests
    cpu: "500m"      # 2x requests

üîç Health Check Mistakes

Mistake 7: Missing Readiness Probes

‚ùå Wrong:

containers:
- name: app
  image: my-app:1.0
  # No readiness probe

Why it's wrong:

  • Traffic sent to Pods before ready
  • 503 errors during startup
  • Poor user experience

‚úÖ Correct:

containers:
- name: app
  image: my-app:1.0
  readinessProbe:
    httpGet:
      path: /ready
      port: 8080
    initialDelaySeconds: 5
    periodSeconds: 5

Mistake 8: Wrong Probe Paths

‚ùå Wrong:

livenessProbe:
  httpGet:
    path: /health  # Doesn't exist
    port: 8080

Why it's wrong:

  • Probe fails ‚Üí Pod restarts
  • Application crashes unnecessarily

‚úÖ Correct:

livenessProbe:
  httpGet:
    path: /healthz  # Actual endpoint
    port: 8080
  initialDelaySeconds: 30
  periodSeconds: 10

Mistake 9: Too Aggressive Probe Timing

‚ùå Wrong:

livenessProbe:
  httpGet:
    path: /health
    port: 8080
  initialDelaySeconds: 1  # Too short
  periodSeconds: 1        # Too frequent

Why it's wrong:

  • Pods killed before ready
  • Unnecessary restarts
  • Resource waste

‚úÖ Correct:

livenessProbe:
  httpGet:
    path: /health
    port: 8080
  initialDelaySeconds: 30  # Allow startup time
  periodSeconds: 10        # Reasonable interval

üîê Configuration Mistakes

Mistake 10: Hardcoding Configuration

‚ùå Wrong:

env:
- name: DB_HOST
  value: "production-db.example.com"
- name: API_KEY
  value: "secret-key-123"

Why it's wrong:

  • Different values per environment
  • Secrets in YAML files
  • Hard to manage

‚úÖ Correct:

env:
- name: DB_HOST
  valueFrom:
    configMapKeyRef:
      name: app-config
      key: db_host
- name: API_KEY
  valueFrom:
    secretKeyRef:
      name: app-secrets
      key: api_key

Mistake 11: Committing Secrets to Git

‚ùå Wrong:

# secrets.yaml committed to Git
apiVersion: v1
kind: Secret
data:
  password: c2VjcmV0MTIz  # Base64 encoded (easily decoded!)

Why it's wrong:

  • Secrets exposed in version control
  • Security risk
  • Hard to rotate

‚úÖ Correct:

  • Use external secret managers (Vault, AWS Secrets Manager)
  • Use CI/CD secret injection
  • Never commit secrets

Mistake 12: Using default Service Account

‚ùå Wrong:

# Not specifying serviceAccountName
spec:
  containers:
  - name: app
    image: my-app:1.0

Why it's wrong:

  • Default SA may have excessive permissions
  • Security risk
  • No least privilege

‚úÖ Correct:

spec:
  serviceAccountName: my-app-sa
  containers:
  - name: app
    image: my-app:1.0

Mistake 13: Using Pods Directly

‚ùå Wrong:

apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
  - name: app
    image: my-app:1.0

Why it's wrong:

  • No self-healing
  • No scaling
  • No rolling updates

‚úÖ Correct:

apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
      - name: app
        image: my-app:1.0

Mistake 14: Using latest Tag

‚ùå Wrong:

image: my-app:latest

Why it's wrong:

  • Unpredictable deployments
  • Hard to rollback
  • Version tracking issues

‚úÖ Correct:

image: my-app:v1.2.3  # Specific version

Mistake 15: Not Using Namespaces

‚ùå Wrong:

# Everything in default namespace
kubectl get all

Why it's wrong:

  • No organization
  • Resource conflicts
  • Hard to manage

‚úÖ Correct:

# Use namespaces
kubectl create namespace production
kubectl create namespace staging
kubectl create namespace development

üîí Security Mistakes

Mistake 16: Skipping RBAC

‚ùå Wrong:

# Everyone has cluster-admin
kubectl create clusterrolebinding admin-binding \
  --clusterrole=cluster-admin \
  --user=everyone

Why it's wrong:

  • No access control
  • Security risk
  • Violates least privilege

‚úÖ Correct:

# Create specific Role
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: developer
rules:
- apiGroups: [""]
  resources: ["pods", "services"]
  verbs: ["get", "list", "create"]

Mistake 17: Running as Root

‚ùå Wrong:

# Not specifying security context
containers:
- name: app
  image: my-app:1.0

Why it's wrong:

  • Security risk
  • Container escape risk

‚úÖ Correct:

spec:
  securityContext:
    runAsNonRoot: true
    runAsUser: 1000
  containers:
  - name: app
    image: my-app:1.0
    securityContext:
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem: true

üíæ Storage Mistakes

Mistake 18: Using emptyDir for Persistent Data

‚ùå Wrong:

- name: data
  emptyDir: {}

Why it's wrong:

  • Data lost when Pod deleted
  • Not persistent

‚úÖ Correct:

- name: data
  persistentVolumeClaim:
    claimName: my-pvc

Mistake 19: Not Setting StorageClass

‚ùå Wrong:

# No storageClassName

  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 10Gi

Why it's wrong:

  • May use slow storage
  • No dynamic provisioning

‚úÖ Correct:

spec:
  storageClassName: fast-ssd
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 10Gi
